## 1. 🔴 Always-on attester deployment

**What:** A dedicated long-running proxy instance (or k8s pod) that produces a continuous feed
of attestations to the Sepolia (then mainnet) `VaultReputation` contract. Without this, the
"reputation primitive" story is aspirational — `vault inspect` queries an emptier contract.

**Why deferred:** v0.2 ships before this exists. We'd be marketing infra we don't run. Better
to launch as a pure security tool and ship the attester once we have real install traffic
to attest *about*.

**Pre-conditions before starting:**
- Real install traffic from v0.2.0-rc.1 going to at least 50 distinct MCP servers (~1 week
  post-launch, depending on uptake).
- A hosted MCP server we control + light synthetic traffic generator (so the attester has
  something to scan).
- Funded Sepolia hot wallet (~0.05 ETH).
- Mainnet contract deploy (item #2).

**Estimated effort:** 2–4 hours engineering + ~1 day of soak testing. The proxy already does
attestation; this is config + pm2 process + monitoring.

**Risks / open questions:**
- Hot-wallet operational security: rotate weekly? key custody?
- Cost projection on mainnet: at attestation rate X per hour, ETH/month spend = Y. Need to
  model before mainnet flip.
- What's the synthetic traffic source? Don't want to attest to attacks we made up; need
  genuine traffic against a real MCP server.

## 2. 🔴 Mainnet contract deploy

**What:** Deploy `VaultReputation.sol` to Base mainnet (chain 8453). Set the production
attester address in the allowlist. Update `packages/contracts/deployments.json` and the
proxy's Sepolia-default to point at mainnet.

**Why deferred:** No always-on attester yet (item #1). Deploying a public mainnet contract
nobody writes to is just expensive on-chain noise.

**Pre-conditions:**
- Item #1 attester is running on Sepolia and has produced at least 7 days of clean writes.
- Wallet funded with ~0.05 ETH on Base mainnet for deploy + early txs.
- `VAULT_OWNER` address decided (msig recommended).
- `forge verify-contract` workflow ready.

**Estimated effort:** ~2 hours including Basescan verification + initial allowlist tx +
demo-site env var rollout.

**Risks:**
- Contract is `transferOwner`-able but not upgradeable. Bug = deploy v2 + migration script.
  Re-audit the contract before deploying.
- ETH cost is operational forever, not one-shot.

**Fix (~10 min):**

3. `pnpm install` to regenerate the lockfile.
4. Verify `pnpm -F @vaultmcp/eval typecheck` still works.

**Risks:** none — eval package is dev-only, doesn't ship to users.

## 7. 🟢 Switch publish workflow from pnpm publish to npm publish

**What:** v0.2.0-rc.1 was published via `pnpm publish --no-git-checks`, which leaves
`gitHead` empty in the npm registry. Switch to `npm publish` (or `pnpm publish` with
`--git-checks`) for v0.3 so the provenance link is machine-readable.

**Fix (~5 min):** update the release script (or memorize: use `npm publish` instead).

**Risks:** none.

## 8. 🟢 Deprecated transitive deps monitoring

**What:** `prebuild-install@7.1.3` and `node-domexception@1.0.0` are deprecated transitive
dependencies surfaced by the v0.2 install. Both are upstream — we don't control them
directly. Watch for breaking-version bumps over the next 6 months.

**Action:** quarterly `npm audit` + `pnpm outdated --recursive` review. If either breaks,
the fix is to update the parent dep (likely `better-sqlite3` for `prebuild-install`).

**Risks:** if `better-sqlite3` ships a major version that drops `prebuild-install`, we may
need to switch persistence to `node:sqlite` (Node 22.5+). Tracked in v0.4 notes.
